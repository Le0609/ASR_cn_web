# 技术债务与优化计划

本文档记录 Echo v0.0.1 当前已知的技术债务、待优化问题和改进方向，供后续版本迭代参考。

---

## 🔴 高优先级（影响用户信任和体验）

### 1. 置信度指标显示虚假数据

**现状**：
- 前端 `App.tsx:65-75` 使用 `Math.random()` 生成假置信度（快速模式 70-82%，精准模式 85-95%）
- `StatusBar.tsx` 显示为绿/黄/红指示灯，暗示"识别质量"
- 后端实际从未返回置信度数据

**问题**：
- 快速模式实际 CER ~29%（近三成字错），但界面显示 ~76% 绿灯
- 误导用户对识别质量的判断，影响专业场景下的信任度
- 虚假数据比"无数据"更危险

**建议方案**：
1. **方案 A（推荐）**：完全隐藏置信度行，只保留"处理耗时"等真实指标
2. **方案 B**：改为显示"模型类型"（Conformer / FunASR）而非置信度
3. **方案 C**：后端实现真实置信度计算（需修改 Modal ASR 服务）

**影响范围**：
- `src/App.tsx:65-75` - `generateConfidenceForMode()` 函数
- `src/App.tsx:286-293, 357-366` - 置信度赋值逻辑
- `src/components/StatusBar.tsx:20-26` - 指示灯颜色判断

---

### 2. 隐私声明与实际架构矛盾

**现状**：
- `PrivacyBanner.tsx` 宣称"端到端加密""本地推理处理""仅浏览器缓存"
- `Footer.tsx:30` 写"音频与转写数据仅存储在你的浏览器本地，不会上传到任何服务器"
- 实际：音频通过 HTTP POST 上传到 Modal 云端推理

**问题**：
- 属于虚假宣传，可能违反 GDPR/个人信息保护法
- 用户基于错误信息做出的授权无效
- 影响产品合规性和用户信任

**建议方案**：
1. **方案 A（推荐）**：改文案为真实描述
   ```
   - "转写数据本地存储"（IndexedDB 是真的）
   - "音频加密传输"（如果后端配了 HTTPS）
   - "服务器不保留音频"（如果确实如此）
   ```
2. **方案 B**：删除整个 `PrivacyBanner` 组件，在 Footer 简要说明
3. **方案 C**：实现真正的本地推理（需要 WebAssembly + ONNX Runtime，工程量大）

**影响范围**：
- `src/components/PrivacyBanner.tsx` - 全文需改写
- `src/components/Footer.tsx:30` - 改为"转写数据仅存储在你的浏览器本地"

## 🟡 中优先级（功能完整性和稳定性）

### 3. 并发请求会阻塞事件循环

**现状**：
- 后端 `api_server_final.py` 的 `transcribe_fast/accurate` 是 `async def`
- 但内部调用同步阻塞的 CPU 推理（`transcribe_conformer()`）和 `subprocess.run(ffmpeg)`
- 单进程 uvicorn 下，一个请求处理时会卡住其他请求（包括 `/health`）

**问题**：
- 多用户同时转写时，后来的请求会长时间等待
- 健康检查也会被卡住，前端误判为"后端离线"

**建议方案**：
1. 使用 `asyncio.to_thread()` 或 `run_in_executor()` 将同步推理丢进线程池
2. 或启动多个 uvicorn worker（`--workers 4`）
3. 或改用 Gunicorn + Uvicorn workers

**影响范围**：
- `ASR/api_server_final.py:172-189` - `transcribe_conformer()` 调用
- `ASR/services/funasr_service.py:140-189` - FunASR 推理调用

---

### 4. 首次精准模式加载体验差

**现状**：
- FunASR 模型首次调用需下载/加载 1-2GB，耗时 20-80 秒
- 前端进度条按 3 秒估算，会卡在 98% 长达一分钟
- 用户可能误以为卡死，刷新页面导致重新下载

**问题**：
- 用户体验极差，可能导致首次使用即放弃

**建议方案**：
1. **方案 A（推荐）**：后端增加 `/api/warmup` 端点预热模型，前端首次显示"正在准备模型，请稍候..."
2. **方案 B**：前端检测首次精准模式请求超过 10 秒，动态调整进度条速度
3. **方案 C**：Modal 部署时用 `@app.function(keep_warm=1)` 保持一个实例常驻（需付费）

**影响范围**：
- `ASR/services/funasr_service.py:140-189` - 模型加载逻辑
- `web_ASR/src/api/realASR.ts` - 前端进度估算

---

### 5. 环境变量配置不够灵活

**现状**：
- `VITE_ASR_ENDPOINT` 只支持单个后端地址
- 如果后端有多个区域部署（如国内/海外），无法自动选择最快的

**建议方案**：
1. 支持多个后端地址，前端启动时自动 ping 选择最快的
2. 或根据用户 IP 地理位置自动路由
3. 或提供 UI 让用户手动切换

**影响范围**：
- `src/api/asr.ts` - 初始化逻辑
- 新增 `src/utils/endpointSelector.ts`

## 🟢 低优先级（优化和增强）

### 6. 缺少自动化测试

**现状**：
- 前端零测试文件（无 `.test.ts/tsx`）
- 后端只有 3 个基础测试（`tests/test_postprocess.py` 等）

**建议方案**：
1. 前端：Vitest + React Testing Library，覆盖核心组件
2. 后端：pytest + 完整的接口测试
3. E2E 测试：Playwright 覆盖关键流程

**影响范围**：
- 新增 `web_ASR/src/**/*.test.tsx`
- 新增 `ASR/tests/test_api.py`

---

### 7. 无版本控制历史

**现状**：
- 项目从 v1 → v2.0（批量导出、IndexedDB、重试、预处理）全程无 git commit
- 无法回滚，无法审查 diff

**建议方案**：
1. 立即 `git init` 并推送到 GitHub（已在部署流程中）
2. 后续每个功能都创建 feature 分支 + PR

---

### 8. 缺少鉴权和限流

**现状**：
- 后端 CORS 允许所有来源（`allow_origins=["*"]`）
- 无 API Key、无请求频率限制
- 任何人都能直接调用 Modal API

**问题**：
- Modal 免费额度 $1/月，恶意刷量会耗尽
- 可能被第三方网站盗链

**建议方案**：
1. **短期**：后端添加简单的 API Key 验证（通过 Header 传递）
2. **长期**：使用 Redis + 滑动窗口限流（每 IP 每小时最多 N 次）
3. **Modal 层面**：配置 Secrets 环境变量存储密钥

**影响范围**：
- `ASR/api_server_final.py` - 添加 `Depends(verify_api_key)` 中间件
- `web_ASR/src/api/realASR.ts` - 请求头添加 `X-API-Key`

---

### 9. 前端性能优化空间

**建议**：
1. 使用 `React.lazy()` + `Suspense` 按需加载组件
2. IndexedDB 操作改为 Web Worker（避免阻塞主线程）
3. 音频预处理放到 Web Worker

---

### 10. 用户体验细节

**待优化**：
1. 历史记录超过 50 条时，增加分页或虚拟滚动
2. 批量导出 ZIP 时，显示每个文件的打包进度
3. 音频播放器增加快进/快退（5秒跳转）
4. 编辑器支持查找/替换
5. 支持多语言（英文界面）
